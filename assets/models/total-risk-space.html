Fig. 1 The total risk space of a model M. Each point in the cube is one configuration the system could be evaluated under — an Input (x), against a Version (v), in an Environment (ω) — and the cloud is a stress sweep across that space. It's the same move banks use to value an option: stress the underlying across every plausible state, log the response, read the shape. The cube can be populated from production call logs, from deliberate stress scenarios, or from both. Environment (ω) is the runtime environment, decomposing into ωdata, ωcompute, and ωconfig — each evolving on its own governance cadence (data refreshes fastest, compute slowest, config whenever someone deliberately changes it). How to read the cube
Notes

The cloud. Each point is one configuration the model could be evaluated under: an Input (x), against a Version (v), in an Environment (ω). Some points come from production call logs. Others come from deliberate stress scenarios run by a model risk team to probe corners of the space production has not reached. Together they map the response surface. Where the cloud is dense the model has been thoroughly probed; where it is empty the model has no evidence of how it would behave, and any change that would push traffic there is, by definition, a governance event.

Vertical · Version (v). Read the cube vertically and you read the version axis. v is discrete because versions are discrete: each row in the cube collects every evaluation at one fixed version. A nearly empty row means a version that was deployed briefly. A dense row means a version that has carried most of the load, or that a stress team has probed most aggressively.

Depth · Environment (ω). Read the cube along its depth and you read ω, the runtime environment: everything the model needs to run that isn't the input. ω decomposes into three components: ωdata (feature stores, retrieval corpora, market feeds), ωcompute (hardware, OS, container image), and ωconfig (thresholds, flags, system prompts). Each evolves on its own governance cadence. The split is the place where governance lives — a different team owns each component, on a different schedule, and the boundary between them is where a change has to clear an explicit review.

Walls · the shadows. Read the walls and you read what the cube looks like from each direction with one axis collapsed. They are shadows of the cloud — the same evaluations projected onto each pair of axes. Input × Version on the back wall; Input × Environment on the floor; Version × Environment on the side wall.

How to read. x is the horizontal axis: continuous inputs. v is the vertical axis: one row per discrete model version. ω is the depth axis: the discrete environments the system has been evaluated under. Each point is one configuration evaluated, drawn from production call logs, deliberate stress scenarios, or both. The walls are shadows of those same points projected onto each pair of axes.
